## The 30-second version

Agents fail constantly, and treating every failure the same way is itself a failure mode. The first job of error handling is classification: is this a **transient failure** — a flaky network call, a momentary rate limit, the kind of thing that succeeds if you just try again — or a **logic error** — wrong arguments, a stale assumption, a plan built on a wrong premise, the kind of thing that fails identically every time you repeat it? Transient failures get a retry with backoff; logic errors need a self-correction loop that revises the plan, not the API call. The failure mode above both of these is the one that actually does the damage: an agent that "recovers" by taking a confident, unverified action that leaves things worse than doing nothing would have. Good recovery design spends as much effort bounding how bad a bad recovery can be as it spends maximizing how often recovery succeeds.

## The analogy

You're a bike courier partway through a route, ten packages still to deliver before the cutoff.

You turn onto a street and find it closed for a parade. This is a **transient failure**: nothing about your route was wrong, the world just changed underneath you for an hour. The right move is simple — wait it out, or take the very next parallel street and rejoin your route a block later. Either way, the delivery still succeeds, and it costs you a few minutes.

Now compare a wrong address on the label. You bike to the building, and no such unit exists. Turning around and riding to the exact same address again changes nothing — this isn't a "try again" problem, because repeating the same action produces the same wrong outcome every time. What actually needs to happen is checking the label against the customer's account, or pulling up the order — revising your understanding of where you're supposed to go, not repeating the ride. That's a **logic error**.

There's a third moment, and it's the one that actually costs you the day. Running late, you spot a narrow alley that looks like a shortcut back to the main road. You duck in, confident it saves five minutes — and it dead-ends at a construction fence you didn't see from the entrance. Now you're not just as late as you would have been; you're later, because backing a loaded bike out of a dead end costs more time than the longer route you passed up would have. That's the failure mode worth fearing most: not the closed street, not the wrong address, but the "recovery" you were sure would help, taken with too much confidence and too little verification, that leaves you worse off than if you'd done nothing at all.

The dispatcher exists for exactly the case where none of your own moves are working: after the same block fails you three times in a row, you radio in instead of improvising a fourth variation on your own.

| Bike delivery | Error handling and recovery |
|---|---|
| Street closed for a parade | Transient failure — retry after backoff, or reroute |
| Wrong address on the label | Logic error — retrying the same call fails identically; the plan needs revising, not repeating |
| Checking the label against the account before re-riding | Self-correction loop — feed the failure back and revise the plan before re-attempting |
| Ducking into a shortcut alley that dead-ends | A recovery action taken without verification, leaving the agent worse off than before |
| Backing a loaded bike out of a dead end | The cost of undoing a bad recovery — often higher than the cost of the original failure |
| Radioing the dispatcher after three failed attempts on one block | Circuit breaker — stop retrying blindly and escalate |
| Confirming the package matches the door before marking it delivered | Output validation — catching a confident, silent wrong answer before it's trusted as success |

## How it actually works

![A failure is classified as transient or logic first; logic errors get a verified self-correction loop instead of a blind retry, and repeated identical failures trip a circuit breaker instead of a fourth attempt](diagrams/error-handling-and-recovery--failure-classification.svg)

Follow the diagram from a failed tool call. The first fork is classification, not retry: **is this transient?** — a timeout, a rate limit, an environment blip with no relationship to whether the agent's plan was correct. If yes, retry with exponential backoff (1 second, then 2, then 4) up to a small cap, because the same call has a real chance of succeeding once the underlying condition clears. If the failure is a **logic error** instead — the tool exists but the arguments were wrong, or the call succeeded against the wrong premise — retrying the identical call is pointless; it will fail identically every time. What has to happen instead is a **self-correction loop**: the error gets fed back into the model's context as information ("this call failed because X"), and the model revises the plan before the next attempt, not just the parameters.

The step most systems skip is what sits between "the model produced a recovery" and "the recovery counts as done": **output validation**. A recovery attempt that looks plausible isn't automatically correct, and treating it as correct without checking is exactly how an agent turns one failure into two. A cheap verifier — sometimes just a smaller model, sometimes a deterministic check — confirms the recovery actually resolves the original problem before the agent moves on. If validation fails, the right move is not to layer a third guess on top of a second one; it's to roll back to the last known-good checkpoint (see [Durable Execution](./durable-execution.mdx)) and try a materially different approach, or stop.

Finally, both paths share a counter. If the exact same `(tool, arguments)` pair fails three times in a session — whether from repeated transient bad luck or from a self-correction loop that keeps converging on the same wrong idea — that pattern itself is the signal. A **circuit breaker** trips, the agent is blocked from trying that exact action again, and the task routes to a human or a simplified fallback path (see [Human-in-the-Loop Patterns](./human-in-the-loop-patterns.mdx)) instead of grinding through a fourth attempt that the first three already told you would fail.

## A concrete example

**Retry economics.** A pipeline agent makes 500 external API calls a day. 4% (20 calls) hit a transient `429 Too Many Requests`; retried with backoff (1s, 2s, 4s), essentially all 20 succeed within the first or second attempt, adding about 2.3 seconds of latency each and no meaningful cost. Separately, 1.5% (about 8 calls a day) fail with a genuine schema violation — wrong argument shape for the tool. If those 8 were blindly retried the same way as the transient failures, each would fail identically up to 3 times before giving up: 8 × 3 = 24 wasted calls a day, each carrying a full model round-trip (~$0.002 and ~800 ms). At this scale that's under a dollar a month — trivial. Run the same pipeline at 50,000 calls a day instead of 500, and the same 1.5% logic-error rate blindly retried becomes 2,250 wasted calls a day, or roughly $135 a month and 30 minutes of pure wasted latency, for zero additional success — because a logic error retried blindly never succeeds no matter how many times you repeat it. Classifying the failure and routing schema violations to self-correction instead of blind retry recovers most of that cost and, more importantly, actually fixes some of those 8 calls instead of just re-failing them politely.

**The cost of a bad recovery.** A nightly job reconciles duplicate customer records across 12,000 accounts. On one run, a stale search index causes 40 lookups to come back empty (transient — the index catches up within hours). Without a recovery-quality floor, the agent's fallback is to fuzzy-match each of those 40 against the nearest similar record and merge automatically. Historical incident data on this exact failure shows about 22% of those fuzzy-matched merges (9 of 40) merge the wrong two customers, corrupting shared order history. Unwinding one bad merge takes a support and engineering team about 3 hours; 9 bad merges costs **27 engineer-hours**. The alternative — leaving those 40 records unmerged and letting the next nightly run retry them once the index is current — costs **zero hours**, because the underlying transient condition resolves on its own. The "recovery" was strictly worse than doing nothing.

## The tradeoffs that matter

| Strategy | Fixes transient failures? | Fixes logic errors? | Risk of compounding damage | Cost |
|---|---|---|---|---|
| Blind retry (no classification) | Yes | No — fails identically every time | Low, but wastes calls on unfixable errors | Wasted latency + tokens at scale |
| Classify + retry / self-correct | Yes | Often — the plan actually changes | Moderate — a self-correction can still be wrong | One extra model call to classify and revise |
| + Output validation before accepting | Yes | Yes | Low — a bad recovery gets caught before it's trusted | One verification call per recovery attempt |
| + Checkpoint / rollback on validation failure | Yes | Yes | Lowest — a failed recovery reverts instead of compounding | Requires state snapshots (see durable execution) |
| Circuit breaker after N identical failures | N/A | N/A | Prevents indefinite compounding | Human or fallback-path cost |

The load-bearing layer is output validation. Retries and self-correction both increase the *chance* of recovering; only validation catches the case where the "recovery" was confidently wrong, which is the one scenario that actively makes the system worse rather than merely failing to help.

## Where people go wrong

- **Retrying every failure the same way.** A schema violation retried like a network blip fails identically every time and just burns latency and tokens for nothing.
- **Trusting a recovery attempt without checking it.** A plausible-looking fix isn't a verified fix; skip validation and you risk turning one failure into a worse, silent one.
- **No cap on retries or self-correction attempts.** Without a circuit breaker, a logical stall repeats indefinitely — the classic "loop of death" that burns cost and time until someone notices.
- **No rollback path when a recovery fails validation.** Without a checkpoint to fall back to, the next move is another guess layered on the last one, and each layer makes the eventual cleanup more expensive.
- **Confusing "the tool returned 200 OK" with "the tool succeeded."** A silent failure — the call worked, but the data is wrong — passes right through error handling built only to catch exceptions, not bad answers.

## The interview lens

Interviewers use this topic to see whether you reach past try/catch — whether you know that an agent's failures need diagnosis before treatment, and that a "successful" recovery still needs to be checked before you trust it.

A strong sound bite: *"I classify a failure before I react to it — transient gets a bounded retry, a logic error gets a self-correction loop that revises the plan, and either way the recovery attempt gets validated before I call it done, because an unverified 'fix' that's actually wrong is worse than the original failure."*

Likely follow-ups:

- How do you tell a transient failure from a logic error automatically? (Error type and message often signal it directly — timeouts and rate limits are transient by nature; schema and validation errors are not — and a repeat of the identical failure after one retry is itself evidence it's not transient.)
- What stops a self-correction loop from looping forever? (A counter on identical `(tool, args)` attempts; after a small fixed number, trip a circuit breaker and escalate instead of trying again.)
- How do you catch a recovery that "succeeded" but produced the wrong result? (A verification step — often a cheaper model or a deterministic check — confirms the recovery actually resolves the original problem before the agent proceeds; on failure, roll back to the last checkpoint rather than guessing again.)
