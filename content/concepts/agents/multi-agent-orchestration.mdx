## The 30-second version

A multi-agent system hands one job to several specialized agents instead of one generalist. There are two basic shapes: the **supervisor pattern**, where a coordinating agent breaks the task into pieces, delegates each piece to a worker, and merges the results; and the **hand-off pattern**, where agents pass the task directly to one another, like a chain, with no central coordinator re-planning at every step. Multi-agent design earns its keep when sub-tasks are genuinely independent and can run in parallel, or need different tools and models. It costs you extra round-trips, duplicated context, and new ways to fail — so the real design question is never "how many agents should this have?" It's "does this task actually decompose into pieces that benefit from being separate?"

## The analogy

Picture a small production company shooting a short film in a single week.

If the shoot is simple enough — one person, one location, one static shot — you don't need a crew. One competent person with a camera can direct, shoot, and edit. That's your single agent: fine for small, self-contained jobs.

Once the shoot spans multiple locations, a cast, sound, and a hard delivery date, one person can't do it all *and* keep up quality. So you bring in a crew with a clear structure. The **director** doesn't operate the camera or mix the audio — they read the script, break it into shot lists and daily assignments, and hand each assignment to a specialist: the **cinematographer** frames and lights the shots, the **sound recordist** captures clean audio, the **editor** assembles footage into scenes. Every evening the crew reviews the day's **dailies** together — a checkpoint where the director sees what actually got captured, catches problems early ("the audio from scene 4 is unusable, we need a pickup shot"), and adjusts tomorrow's assignments. That's the **supervisor pattern**: one coordinator decomposes the job, dispatches it, and reconciles the output before it moves downstream.

Contrast that with a **hand-off**. On some productions there's no single director orchestrating every department for every decision. The script supervisor finishes their pass and hands the marked-up script straight to the editor, who works directly from it without routing back through a center. Each specialist trusts the previous one's work and picks up exactly where it left off — faster, with less overhead, but with no one checking that the pieces still add up to a coherent film until the very end.

And sometimes a production needs a specialist it doesn't employ — a visual-effects house on a different continent, running its own tools on its own schedule. The production doesn't fold that house into its internal crew; it sends a shot list and a delivery spec through a standard format and gets finished shots back. That's what crossing an organizational boundary looks like: you don't merge teams, you agree on an interface.

| Film production | Multi-agent system |
|---|---|
| One-person shoot, one location | Single agent handling a self-contained task |
| The director | The orchestrator / supervisor agent |
| Cinematographer, sound recordist, editor | Specialized worker agents |
| Shot list handed to each department | Sub-task delegated to each worker |
| Evening dailies review | Checkpoint where the orchestrator reconciles worker output |
| Script supervisor handing pages straight to the editor | Hand-off pattern — no central coordinator between steps |
| External VFX house on its own systems | Cross-vendor agent reached over a standard protocol (A2A, Agent-to-Agent) |
| A shot list and delivery spec instead of shared internal tools | Agent Cards / capability discovery instead of a shared framework |

## How it actually works

![Two ways to split work across agents: a supervisor that delegates and aggregates, versus a hand-off chain where agents pass work directly to each other](diagrams/multi-agent-orchestration--topology-comparison.svg)

The diagram lines up the same five steps on both sides so you can compare them directly.

On the left, the **supervisor pattern**: a request arrives, and the orchestrator's first job is decomposition — turning "handle this" into a set of concrete, independently answerable sub-tasks. It dispatches each sub-task to a worker, often a different and cheaper model than the orchestrator itself (a high-reasoning model for planning, fast models for execution). When workers finish, the orchestrator's second job is aggregation: reconcile the results, check they actually answer the original request, and produce the final response. Two failure modes live here. **Decomposition failure** — if the orchestrator splits the task into pieces that are logically inconsistent or have a hidden dependency it didn't account for, workers confidently produce correct answers to the wrong questions. **Context dilution** — if every worker's full trace gets appended to shared state, the orchestrator's own context fills with noise and it loses track of the original goal. The fix for the first is having workers confirm a sub-task is well-formed before they start; the fix for the second is passing workers only what they need and summarizing their output before it re-enters shared state.

On the right, the **hand-off pattern**: instead of a hub reporting back to a center, each agent finishes its piece and passes the task — and the context accumulated so far — directly to the next agent. There's no round-trip to a coordinator between hops, which removes a source of latency and a single bottleneck. The cost is that no one checks the shape of the whole job until the last agent finishes; a bad hand-off early in the chain propagates all the way through.

A third shape is worth naming even though it isn't pictured: when the sequence isn't fixed — when a step might need to loop back to an earlier one, or branch depending on what it finds — plain chains and simple supervisor loops both break down. Model the workflow as a graph with typed state and conditional edges instead, so a failed step can route back to an earlier node rather than crashing the whole run. See [Planning and Decomposition](./planning-and-decomposition.mdx) for how that decomposition step is actually built.

Finally, both patterns above assume the agents share a framework and a runtime. When an orchestrator needs to delegate across a team or company boundary — a compliance check owned by a different group, a third-party specialist service — it can't assume a shared framework. That's the case for an interoperability protocol: the remote agent publishes a capability description, the orchestrator sends it a structured task over the network, and results come back the same way. It costs real latency (a network hop, not a function call) in exchange for not having to rebuild someone else's agent inside your own stack.

## A concrete example

Take "research a competitor's pricing page, summarize it, and draft a one-paragraph comparison email."

Run this on a single agent: fetch the page and read it (~3,000 tokens of context), summarize it (~700 tokens out), draft the email (~300 tokens out) — three sequential model calls on one thread, roughly 1.5–2 seconds each including tool latency, for a total of about 5 seconds wall-clock and around $0.01 at typical mid-tier pricing.

Now hand it to a supervisor with three workers — a researcher, a summarizer, a writer. The catch: the summarizer needs the researcher's output, and the writer needs the summarizer's output. There's no independent work to parallelize; the dependency chain is identical to the single-agent version. What you add is a planning call for the orchestrator to decompose the task (~300 ms, ~$0.001), an aggregation call to check the final draft (~400 ms, ~$0.001), and three separate system prompts repeated across three separate workers instead of one. Total wall-clock climbs to roughly 6.5–7 seconds and cost rises 20–30% — for a task that finishes with the exact same email. That's coordination overhead with nothing to show for it.

Now take a task shaped differently: "review the last 12 support tickets from the last hour and flag which ones need escalation." Each ticket is an independent, near-identical sub-task — genuinely parallelizable. One agent working through them sequentially, at roughly 1.2 seconds per ticket, takes about 14.4 seconds. A supervisor fanning the 12 tickets out across 6 parallel workers (2 tickets each) finishes in roughly 2 × 1.2 s plus dispatch and aggregate overhead (~0.3 s + 0.3 s) ≈ 3 seconds — a 4–5x wall-clock improvement. Total tokens run somewhat higher (each worker repeats a short system prompt, and the orchestrator adds its own dispatch and aggregate calls — call it 25–30% more tokens than the single-agent run), but for a latency-sensitive path, trading 25% more cost for a 4x speedup is an easy call. Same architecture, opposite verdict, because the second task actually decomposes into independent pieces and the first one doesn't.

## The tradeoffs that matter

| Shape | Coordination cost | Best for | Main failure mode |
|---|---|---|---|
| Single agent | None | Small, self-contained tasks | Runs out of context or tools on complex jobs |
| Supervisor (hierarchical) | One planning call + one aggregation call, per request | Tasks with real parallel sub-tasks; mixed model needs (cheap workers, expensive planner) | Decomposition failure; context dilution in shared state |
| Hand-off (chain) | Lower — no round-trip to a center | Sequential pipelines where each stage's job is unambiguous | No one checks the whole job until the last hop; errors compound silently |
| Graph-based (conditional, with loops) | Higher upfront design cost, lower re-run cost | Non-linear tasks needing retries, branches, or backtracking | Harder to reason about statically; needs good observability |
| Cross-vendor (A2A-style) | Network hop + serialization, per call | Crossing team, company, or vendor boundaries | Latency; trusting internals you don't control |

The number that should drive the decision is how much of the task is actually parallel or genuinely separable by concern. If the sub-tasks form a strict dependency chain, a supervisor buys you organization and evaluability, not speed — you're paying coordination cost for free. If the sub-tasks are independent, that same coordination cost buys you a wall-clock win that scales with how many you can run at once.

## Where people go wrong

- **Adding agents to a task that's a strict dependency chain.** If step 2 always needs step 1's output, splitting it across agents adds a supervisor round-trip and buys you nothing but line items in a diagram.
- **Letting shared state become a dumping ground.** Appending every worker's full reasoning trace to global state feels safe ("nothing gets lost") but it's how an orchestrator loses track of the original goal in its own context window.
- **Dispatching sub-tasks without checking they're well-formed.** A worker given an inconsistent or underspecified sub-task will still produce a confident, wrong answer — it has no way to know the plan itself was broken.
- **Reaching for a cross-vendor protocol inside a single team.** If one team owns every agent and they share a runtime, a network-based interoperability layer only adds latency and serialization you don't need.
- **Treating "multi-agent" as a quality upgrade by default.** More agents means more places for a decomposition to go wrong, not an automatic accuracy boost — it buys parallelism and specialization, not correctness.

## The interview lens

Interviewers ask about multi-agent orchestration to see whether you reach for it because a task's shape demands it, or because it sounds sophisticated.

A strong sound bite: *"I only split a task across agents when the sub-tasks are actually independent or need different tools — otherwise I'm paying for a supervisor's planning and aggregation calls without unlocking any parallelism, and a single agent with a good plan does the same job for less."*

Likely follow-ups:

- How do you stop a supervisor's shared state from ballooning as workers report back? (Summarize before it re-enters shared context; don't append raw traces.)
- When would you pick a graph-based orchestrator over a straight supervisor loop? (When the task needs conditional branches or has to backtrack to an earlier step on failure — a static chain or single loop can't route around that.)
- When does it make sense to cross a vendor or team boundary instead of keeping everything in one framework? (When you don't own the specialist agent and can't fold it into your runtime — accept the network hop for vendor neutrality and independent scaling.)
