## The 30-second version

An embedding model converts text into a fixed-length vector — a list of numbers where similar meaning lands nearby in space. In RAG (Retrieval-Augmented Generation), you embed every chunk at index time and embed each query at search time, then retrieve by nearest-neighbor lookup. The production frontier has moved beyond "one float32 vector per chunk": **Matryoshka** models nest coarse and fine resolution in one embedding; **late-interaction** models like ColBERT store a vector per token; **multimodal** embedders put text, images, and PDF pages in one space. Your embedding choice sets the ceiling on retrieval — swap models and you re-embed the entire corpus.

## The analogy

Think of a library where every book is shelved by **topic coordinates** instead of author name.

A **bi-encoder** librarian reads each book once, writes a single shelf label summarizing the whole thing — "thermal management, GPUs, datacenter" — and files it. When you ask a question, they write a similar one-line label for your question and walk to the nearest shelf. Fast, but a label that says "GPU setup" might sit next to both "CUDA env vars" and "buying a graphics card."

A **Matryoshka** librarian uses the same label format, but the **first few words** on the label carry most of the meaning. For a quick scan they read only the first word; for a disputed match they read the full label. Same filing system, two speeds.

A **ColBERT** librarian files **every sentence** on its own mini-label, then at query time compares your question word-by-word against each sentence label. Painstaking to file, expensive to search — but unbeatable when the answer is one needle sentence in a haystack manual.

| Library habit | Embedding approach |
|---|---|
| One summary label per book | Bi-encoder (single vector per chunk) |
| Read only the bold first words for speed | Matryoshka / MRL (Matryoshka Representation Learning) |
| Label every sentence separately | Late interaction (ColBERT family) |
| Shelve photos and text together | Multimodal embeddings |
| Re-label every book when you change librarians | Mandatory re-embed on model swap |

## How it actually works

![Bi-encoder stores one full vector per chunk; Matryoshka nests coarse-to-fine dimensions in the same vector](diagrams/embedding-models--biencoder-matryoshka.svg)

The left panel is standard RAG indexing. Each chunk passes through an **encoder** (a transformer trained for similarity, not generation) and becomes one vector — commonly 768–3,072 dimensions of `float32`. Vectors are stored in an approximate nearest-neighbor (ANN) index (HNSW — Hierarchical Navigable Small World — is the usual structure). At query time the question is embedded with the **same model** and the index returns the closest chunk vectors.

The right panel shows **Matryoshka Representation Learning**: the model is trained so the leading dimensions preserve most of the semantic signal. You can run ANN on only the first 64 dimensions (optionally **binary quantized** for Hamming distance), retrieve a wide candidate set cheaply, then **rescore** the survivors with the full 1,536-dimensional vector. Index RAM drops by an order of magnitude with typically under 2% accuracy loss — the difference between a feasible billion-vector index and a science project.

Beyond the diagram:

- **Late interaction (ColBERT v2, ColPali):** multiple vectors per document (per token or per page image). Query tokens max-similarity-match document tokens. Higher precision, higher storage and latency — now practical with compressed indexes like PLAID.
- **Multimodal:** Cohere Embed v4, Voyage-Multimodal, Gemini Embedding — one space for text and images so schematic search works without brittle OCR pipelines.
- **Quantization:** `int8` or binary embeddings shrink memory 4–32×; essential at scale.

## A concrete example

Corpus: 500,000 support articles, ~400 tokens each after chunking → 500k vectors.

| Config | Vector size | Index RAM (HNSW, rough) | Notes |
|---|---|---|---|
| 1536-d float32 bi-encoder | 6 KB / vector | ~3 GB vectors + graph overhead | Baseline |
| Same + Matryoshka 64-d binary pass | ~8 B / vector for stage 1 | Stage-1 index ~50× smaller | Rescore top 1,000 with full vec |
| ColBERT-style (128 tokens × 128-d) | ~64 KB / doc | 10×+ storage vs bi-encoder | Wins on needle-in-haystack technical queries |

Embedding cost (one-time index): 500k × 400 tokens ≈ 200M tokens. At ~$0.02/M tokens for a small embedding API → about **$4** total. The expensive part is not the API bill — it is **index RAM and the re-embed migration** when you change models.

Query latency: embedding the question ~20–40 ms; ANN lookup ~20–50 ms on a warm index. Late-interaction adds milliseconds per candidate — another reason to funnel through bi-encoder first.

## The tradeoffs that matter

| Model class | Recall quality | Index cost | Query latency | Best fit |
|---|---|---|---|---|
| Small bi-encoder (e.g. 384-d open) | Good | Low | Fast | Prototypes, narrow domains |
| Large bi-encoder (1536-d commercial) | Very good | Medium | Fast | Default production RAG |
| Matryoshka + binary coarse pass | Very good | Low at scale | Fast + rescore step | Million+ vectors |
| ColBERT / page embeddings | Excellent on hard queries | High | Slower | Technical docs, PDFs with diagrams |
| Multimodal | Adds image/page signal | Medium–high | Medium | Manuals, slides, catalogs |

**Selection checklist:** (1) language coverage — multilingual queries need multilingual training; (2) context length — 8k vs 128k matters for long chunks; (3) hosting — API vs self-hosted open weights (BGE-M3, Qwen3-Embedding) for cost at volume; (4) coupling — document and query encoders must match; (5) eval — MTEB scores are a starting point, your labeled query set is truth.

## Where people go wrong

1. **Treating embeddings as interchangeable.** Vectors from different models (or versions) do not live in the same space. Upgrading models means a full re-embed and index rebuild.
2. **Optimizing chunk size while ignoring embedding quality.** A perfect split into a weak vector space still retrieves garbage.
3. **Assuming bigger dimension always wins.** Past ~1,536 dims, gains diminish; RAM and latency rise linearly.
4. **Skipping hybrid when vocabulary drifts.** New product codes and model names hit **vocabulary mismatch** — pair embeddings with BM25 (see [Hybrid Search](./hybrid-search.mdx)).
5. **Indexing without a version pin.** `text-embedding-3-small-2024-01` and a silent API update can shift geometry. Version the model name in your index metadata.

## The interview lens

Interviewers probe whether you understand embeddings as **infrastructure with migration cost**, not a magic `embed()` call.

A strong sound bite: *"The embedding model is a schema decision — change it and you re-index everything — so I pick based on eval on our queries, multilingual needs, and whether we need Matryoshka or multimodal, not leaderboard vanity."*

Likely follow-ups:

- What is vocabulary mismatch and how do you mitigate it? (Hybrid search + cross-encoder reranking.)
- Why Matryoshka for a billion-vector index? (Coarse binary pass shrinks RAM; full-dim rescore on top 1k.)
- Bi-encoder vs ColBERT — when is late interaction worth the cost? (Needle-in-haystack technical retrieval, PDF page search.)
